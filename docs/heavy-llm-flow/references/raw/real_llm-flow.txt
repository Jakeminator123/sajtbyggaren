Så min tolkning är:

Projektet har dokumenterat den stora LLM-arkitekturen, men den praktiska sprinten valde att säkra skelettet först. Sedan har mycket energi gått till preview, follow-up-säkerhet, branch/PR-scope, UI-lanes, trust/contact och andra nödvändiga saker. 3. Varför det känns som att det inte går framåt

Det är inte för att inget finns. Det är för att mycket av det som finns är förberedande.

Nya Sajtbyggaren har en tydlig åttastegsmodell:

Init Prompt
→ Project Input
→ Starter
→ Scaffold
→ Variant
→ Dossier
→ Generation Package
→ Build

Det finns redan i README som operatorflöde.

Engine Run-policy:n säger också att en run har tre faser: understand, plan, build, och att de skriver artefakter som input.json, site-brief.json, site-plan.json, generation-package.json, quality-result.json, repair-result.json, build-result.json och trace.ndjson.

Men när man tittar på vad LLM:en faktiskt får göra i praktiken just nu blir det mycket smalare:

briefModel kan skapa Site Brief.
planningModel kan välja scaffold/variant/dossiers, men inom hårda rails.
codegenModel är extremt smalt: fil-listan är deterministisk och LLM:en får i den slicen bara berika med rationale och riskNotes, inte injicera filer.
Även codegen-filerna byggs deterministiskt i alla paths.

Det är alltså rimligt att upplevelsen känns som:

LLM hjälper till runt kanterna
men den kreativa hemsidebyggaren är inte riktigt inkopplad

Det är också därför du känner: “vi har byggt ett deterministiskt system, och ett enklare LLM-flöde som mest härmar det.”

Det är en ganska korrekt observation.

4. Går variants, dossiers och scaffolds att mappa direkt till JSON-filer?

Ja. Och det är en av de stora styrkorna i nuvarande grund.

Scaffold

Ett scaffold är redan en mapp med flera JSON-kontrakt. load_scaffold_registry() läser exempelvis:

scaffold.json
routes.json
sections.json
selection-profile.json
compatible-dossiers.json
variants/*.json

och returnerar dem som registry-data till planningModel.

För local-service-business beskriver scaffold.json exempelvis sajttypens syfte: lokal tjänstefirma, trust, tjänster, lokal förankring och offert/kontakt-konvertering.

routes.json säger vilka routes som finns: home, tjänster, om oss och kontakt.

sections.json säger vilka sektioner varje route ska ha. Home har exempelvis hero, service-summary, trust-proof, contact-cta och optional sektioner som story, gallery, testimonials, FAQ, reviews och certifications.

Det betyder att det redan finns en direkt bro:

LLM planerar route/section-intent
→ scaffold/sections.json avgör vad som faktiskt får renderas
Variant

Variant är också JSON. nordic-trust har exempelvis tokens för färg, typografi, radius, spacing, motion och tone/vibe.

Det betyder:

LLM kan säga “trygg, lokal, premium men inte lyxig”
→ planning/variant resolver väljer nordic-trust
→ renderer får tokens

Det som fortfarande är lite splittrat är section treatments. En del variant-till-section-treatment-logik ligger i Python-kod i dispatcher.py, inte i variant-JSON. Dispatchern har redan en trestegsordning: operator-pin vinner, sedan variant-default, sedan section-default. Den kommenterar dessutom att en framtida LLM-pick kan läggas framför utan att röra renderers.

Min rekommendation: flytta gradvis mer av detta till variant-JSON eller en section-treatments.json, så LLM-orchestratorn kan välja visuella treatments deklarativt.

Dossier

Dossier är redan mappbart via capability map. capability-map.v1.json säger uttryckligen att capability-sluggen är kontraktet och att dossiern är implementationen. Dossier-ID ska matcha mappnamn under packages/generation/orchestration/dossiers/{soft,hard}/.

Exempel:

contact-form → mailto-contact-form
menu → menu-display
booking → booking-cta
gallery → image-gallery
hours → opening-hours
reviews → reviews-display
location → map-embed
pricing → pricing-table
faq-section → faq-accordion

Ett dossier-manifest finns också som JSON. interactive-game-loop är exempelvis en soft dossier med capability interactive-game, rewritable code fidelity och sammanfattning av vad den ska garantera.

Så ja: du hade rätt i att det är meningsfullt att jobba runt dessa JSON-artefakter. De är exakt rätt lager att låta LLM:en resonera över, men inte fritt förstöra.

5. Den viktiga slutsatsen om “mallmotor” + “LLM-flöde”

Jag skulle formulera systemet så här:

Scaffold = vad sajten får vara
Variant = hur den får kännas
Dossier = vilka extra förmågor den får ha
Sections = vilka byggblock den kan använda
Project Input = vad kunden faktiskt sagt och valt
Site Brief = vad LLM förstår om kunden
Site Plan = hur sajten ska struktureras
Generation Package = arbetsordern till generatorn
OpenClaw = agenten som håller ihop allt, klassar intention och väljer nästa steg

Deterministiska systemet ska inte ersättas. Det ska bli plattformen som LLM-agenten arbetar ovanpå.

En bra mental modell:

Deterministisk grund:
“Jag vet vilka byggblock som finns och hur man bygger utan att krascha.”

LLM-orchestrator:
“Jag förstår vad användaren försöker åstadkomma och vilka byggblock som bör användas.”

Det är kombinationen som gör Sajtbyggaren starkare än både ren mallmotor och fri kodagent.

6. Var OpenClaw bör sitta

Jag håller helt med dig: OpenClaw/Sajtagenten bör ha en given plats, men inte som en vanlig chatt-widget.

Den ska bli orchestratorn ovanför init/follow-up-flödet.

Gamla projektbeskrivningen säger att OpenClaw hade just den kontextförmåga Sajtbyggaren behöver: läsa builder-kontext, genererad kod, skrivbara textfält, senaste meddelanden, aktiv version och filmanifest/full filtext beroende på frågan. Den hade också en policy för lätt kodkontext, manifest eller full kodkontext.

Det är exakt rätt idé, men den ska byggas renare i nya Sajtbyggaren.

Jag skulle inte kalla den bara “chat”. Jag skulle kalla den:

Sajtagent / OpenClaw Orchestrator

Dess jobb:

1. förstå användarens meddelande
2. avgöra om det är fråga, instruktion, ändring, buggrapport, referens, mixed request
3. välja hur mycket kontext som behövs
4. välja om build/preview ska startas eller inte
5. föreslå eller skapa patch plan
6. kalla rätt tools
7. bevara version/DNA
8. verifiera att ändringen faktiskt syntes
9. svara begripligt till användaren
7. OpenClaw ska alltså inte alltid bygga om

Det här är väldigt viktigt utifrån dina klock-exempel.

Dagens follow-up-risk är ungefär:

Användaren skriver något i chatten
→ systemet antar att det är en site-edit
→ kör prompt/build
→ preview adapter startas
→ iframe uppdateras

Men många meddelanden ska inte trigga build.

OpenClaw behöver därför en första klassificering:

type MessageKind =
  | "answer_only"
  | "site_review"
  | "edit_instruction"
  | "component_discovery"
  | "reference_analysis"
  | "bug_report"
  | "multi_intent"
  | "unclear";

Och separat:

type BuildRequirement =
  | "none"
  | "plan_only"
  | "artifact_patch_only"
  | "targeted_rebuild"
  | "full_rebuild";

Det är den sak som saknas i dagens enkla follow-up-modell.

I nuvarande kod är follow-up-intent deterministiskt, litet och whitelist-baserat. Okända prompts blir konservativa för att undvika läckage/regressioner. Intent-värdena är i princip tone-shift, story-emphasize, tagline-update, positioning-shift, no-semantic-change, clarify.

Det är bra som säker V1, men för dina exempel är det för primitivt.

8. Dina klock-exempel som riktig OpenClaw-routing
Exempel A: “vad är klockan?”

Detta ska bli:

{
  "messageKind": "answer_only",
  "editKind": null,
  "buildRequirement": "none",
  "contextLevel": "none",
  "toolCalls": ["time_tool_if_available"]
}

Ingen site patch. Ingen preview. Ingen adapter. Inget iframe-omstart.

Svar: aktuell tid.

Exempel B: “jag vill ha en klocka på andra sektionen till vänster”

Detta är en faktisk ändring.

{
  "messageKind": "edit_instruction",
  "editKind": "component_add",
  "target": {
    "route": "home",
    "sectionOrdinal": 2,
    "position": "left"
  },
  "componentIntent": "clock_widget",
  "buildRequirement": "targeted_rebuild",
  "contextLevel": "artifacts_plus_sections"
}

OpenClaw bör då:

1. läsa aktuell Site Plan
2. mappa “andra sektionen” till faktisk section id
3. kolla om component/capability finns
4. om inte: föreslå ny soft dossier eller inline component-patch
5. skapa patch plan
6. ändra bara relevant artefakt/filer
7. köra quality/smoke
8. uppdatera preview

Inte “släck ner hela sajten och börja om”.

Exempel C: “vilka klockor finns att tillgå att placera någonstans?”

Detta är inte en byggorder. Det är discovery/guidance.

{
  "messageKind": "component_discovery",
  "editKind": "none",
  "buildRequirement": "none",
  "contextLevel": "component_registry"
}

Svar bör vara något i stil med:

Vi kan lägga in:
- enkel digital klocka
- öppettider/status-klocka
- countdown till event/erbjudande
- lokal tid för flera kontor
- dekorativ analog klocka

Vill du placera den i hero, andra sektionen, footer eller kontaktsidan?

Ingen build förrän användaren väljer.

Exempel D: “jag vill ha samma klocka som på aftonbladet.se”

Detta är en reference-analysis request.

{
  "messageKind": "reference_analysis",
  "editKind": "component_add",
  "reference": {
    "url": "aftonbladet.se",
    "object": "clock"
  },
  "buildRequirement": "plan_only_first",
  "contextLevel": "external_reference_plus_current_artifacts",
  "risk": "do_not_copy_exact_design_or_code"
}

OpenClaw bör:

1. hämta/analysera referensen med tool call om tillåtet
2. beskriva vad som menas visuellt/funktionellt
3. skapa en liknande men egen variant
4. fråga om placering om oklart
5. först därefter patcha

Viktigt: den ska inte kopiera Aftonbladets exakta design/kod. Den ska skapa en inspirerad, egen komponent.

Exempel E: “gör sidan mer premium, lägg in en klocka i andra sektionen och ändra inte texterna”

Detta är multi-intent.

{
  "messageKind": "multi_intent",
  "subtasks": [
    {
      "editKind": "visual_style",
      "scope": "global",
      "instruction": "more premium"
    },
    {
      "editKind": "component_add",
      "component": "clock",
      "target": "home.section[2].left"
    },
    {
      "constraint": "preserve_copy"
    }
  ],
  "buildRequirement": "targeted_rebuild"
}

Detta är exakt där en OpenClaw-orchestrator kan bli game changer. Den ska kunna dela upp användarens text i flera deluppgifter, bevara constraints och köra patchar i rätt ordning.

10. Init-flöde: hur det borde se ut

För första generationen:

Fri prompt / wizard
→ OpenClaw intake
→ business facts extraction
→ Site Brief v2
→ Discovery Resolver
→ scaffold/variant/dossier decision
→ content strategy
→ section-level content blocks
→ visual direction
→ Generation Package v2
→ deterministic build
→ quality critic
→ repair pass
→ preview

Här bör LLM:en vara starkast i:

- affärsförståelse
- positioning
- copy
- målgrupp
- trust
- tjänster/produkter
- CTA-strategi
- visuell riktning
- section-prioritering

Men deterministic rails ska fortfarande äga:

- vilka scaffolds som finns
- vilka variants som är tillåtna
- vilka dossiers som kan mountas
- vilka sections som kan renderas
- vilka routes som får skapas
- build/test
- versionering
- file path safety

Det stämmer med coach-anteckningarnas princip: LLM får tolka, skriva, prioritera, förbättra och föreslå; resolver, planning, versionering, validering och quality gates äger beslut som måste vara reproducerbara.

11. Follow-up-flöde: hur det borde se ut

För follow-up ska OpenClaw alltid börja med:

Är detta ens en ändring?

Sedan:

user message
→ OpenClaw Followup Router
→ context level decision
→ target resolution
→ patch plan
→ artifact patch
→ affected files only
→ quality/visible-effect verifier
→ preview update only if needed

Coach-dokumentet säger redan att init och follow-up ska skilja sig. Init är Generation Package -> full generated site, medan follow-up är current snapshot + user message -> followup intent -> patch plan -> changed artifacts -> affected files only.

Det är precis rätt. Det behöver bara bli mycket mer verkligt och agentiskt.

13. Hur variants/dossiers/scaffolds kompletterar detta

Jag skulle låta LLM:en göra detta:

“Den här prompten beskriver en varm familjär frisörsalong i Göteborg.
Rekommenderad struktur: local-service-business.
Rekommenderad variant: warm-craft/family-warmth om den finns.
Rekommenderade sections: hero, service-summary, story, gallery, trust-proof, contact.
Rekommenderad CTA: Boka tid / Kontakta salongen.
Dossier: gallery om bilder finns, booking bara om faktisk bokningsväg finns.
Content blocks: skriv branschspecifik copy.”

Men LLM:en ska inte själv få säga:

“Skapa ett nytt scaffold X”
“Montera en dossier som inte finns”
“Använd checkout/auth bara för att det låter bra”
“Skapa en fake Google-recension”
“Byt starter utan resolver”

Så här kompletterar de varandra:

LLM:
väljer avsikt, vinkel, copy, prioritet, förslag.

Resolver:
validerar scaffold/variant/dossier.

Renderer:
bygger säkert.

Quality:
säger om resultatet är bra.

OpenClaw:
håller kontexten och väljer rätt nästa steg.

16. Min föreslagna implementation i faser
Fas 1 — OpenClaw Router utan filändringar

Bygg bara klassificeringen.

Input:

userMessage
current siteId/runId/version
selected route/section if inspector has focus

Output:

answer_only / edit / component_discovery / reference / bug / mixed / unclear
buildRequirement
contextLevel

Testa med 30–50 prompts, inklusive dina klock-exempel.

Detta kan byggas utan att röra renderern.

Fas 2 — Context Assembler

Bygg verktygen som hämtar rätt kontextnivå:

none
project DNA
artifacts
routes/sections
manifest
selected files
preview/DOM
external reference

Gamla OpenClaw hade just denna styrka: den hämtade lagom kontext beroende på frågan. Det ska tillbaka, men mer formellt.

Fas 3 — Artifact Patch Plan

Låt OpenClaw skapa patch-planer mot artefakter, inte fria filer först.

Exempel:

- update siteBrief.positioning
- update sitePlan.sectionPlan
- update generationPackage.contentBlocks
- update directives.sectionTreatments
- add requestedCapability

Ingen fri kodpatch ännu.

Fas 4 — Content/Visual Blueprint i befintliga artefakter

Utöka site-brief, site-plan och generation-package så de kan bära:

positioning
content blocks
visual direction
section treatments
component requests
quality risks

Detta är där första generationen börjar kännas 8/10 i stället för 5/10.

Fas 5 — Targeted rebuild

När patch-planen påverkar bara en route/section ska systemet inte behöva “börja om”.

artifact patch
→ affected routes/files
→ targeted render
→ quality check for affected route
→ preview refresh
Fas 6 — Quality Critic + Repair

Lägg in en critic som säger:

hero är generisk
service-list är tunn
trust är tom
CTA är oklar
layout känns som standardmall
referenskomponenten syns inte

Sedan en repair-agent som ändrar rätt artefakt.

Quality Gate finns redan som koncept och artefakt. Engine Run-policy:n låser att Quality Gate och Repair ska finnas på centrala platser, inte utspritt som i gamla Sajtmaskin.

Fas 7 — OpenClaw får begränsade file tools

Först när artifact patch fungerar kan OpenClaw få läsa och patcha filer, men bara med strikt plan:

- läs relevant fil
- föreslå diff
- kör validator
- kör quality
- spara ny version

Inte “skriv om projektet”.

18. Finns plan för detta?

Mitt svar är nu mer nyanserat än tidigare:

Ja, det finns en plan på arkitektur-/visionsnivå.
Den finns i LLM Golden Path-dokumenten, Engine Run-policy:n, README, product-operating-context och coach-arkitekturanteckningar.

Nej, den är inte fullt ut aktiv som produktiserad implementation.
Den första LLM Golden Path-sprinten låste befintligt flöde, men byggde inte den tunga orchestratorn. Real codegenModel, Path B och Sajtagent 2.0 ligger uttryckligen utanför den v1-låsningen.

Ja, variants/dossiers/scaffolds är redan bra JSON-ankare för det du vill göra.

19. Min tydliga rekommendation

Jag skulle inte kalla nästa stora steg “mer LLM i codegen”.

Jag skulle kalla det:

OpenClaw Orchestrator V1 + Artifact Patch Flow

Det borde ha tre leveranser:

1. Follow-up Router:
   Klassar answer/edit/review/reference/mixed och avgör buildRequirement.

2. Context Assembler:
   Hämtar rätt artifacts/files/preview/reference beroende på fråga.

3. Artifact Patch Planner:
   Skapar ändringsplan mot Site Brief, Site Plan, Generation Package eller Project Input,
   innan någon full build körs.

När detta finns kan Sajtbyggaren börja kännas som:

“Jag pratar med en agent som förstår sajten”

i stället för:

“Jag skickar promptar till en mallgenerator”

Jag skulle ge en Builder-agent ungefär detta uppdrag:

Bygg OpenClaw Orchestrator V1 för Sajtbyggaren.

Mål:
Skapa ett routing- och planeringslager ovanpå befintligt init/follow-up-flöde som avgör om ett användarmeddelande är:
- answer_only
- site_review
- edit_instruction
- component_discovery
- reference_analysis
- bug_report
- multi_intent
- unclear

V1 ska INTE skriva fria filer och INTE bygga om sajten automatiskt för alla meddelanden.

Input:
- userMessage
- siteId/runId/version om vald
- current Project Input/DNA om tillgänglig
- Site Plan route/section map
- optional selected route/section från UI

Output:
- messageKind
- editKind
- buildRequirement: none | plan_only | artifact_patch_only | targeted_rebuild | full_rebuild
- contextLevel
- target route/section/component om identifierbart
- actionPlan
- rationale
- requiresClarification

Testfall:
1. "vad är klockan" => answer_only, buildRequirement none.
2. "jag vill ha en klocka på andra sektionen till vänster" => edit_instruction/component_add, targeted_rebuild.
3. "vilka klockor finns att tillgå?" => component_discovery, no build.
4. "jag vill ha samma klocka som på aftonbladet.se" => reference_analysis, plan_only first.
5. "gör sidan mer premium och lägg in en klocka men ändra inte texterna" => multi_intent med preserve_copy constraint.

Scope:
- Lägg till router/typer/tester.
- Läs befintliga artifacts men ändra inte build_site.py i denna slice.
- Starta inte PreviewRuntime i answer_only eller plan_only.
- Dokumentera hur detta senare kopplas till Artifact Patch Planner.

Definition of done:
- Router-testsvit grön.
- Inga befintliga follow-up-versioneringstester får regreddera.
- OpenClaw V1 kan förklara varför den inte startar build för rena frågor.
