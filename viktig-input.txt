Viktiga risker/buggar jag skulle fixa först
1. routerModel finns, men används inte där det mest behövs

#188 lägger classify_message_with_llm_fallback ovanpå deterministiska routern och säger att fallback ska användas för unclear, långa eller komplexa multi-intent-meddelanden.

Men i run_followup_chain importeras och används fortfarande classify_message, inte fallback-entrypointen.

Påverkan: i verklig CLI-followup med OPENAI_API_KEY får oklara/långa följdpromptar inte LLM-fallback. KÖR-6b blir mer “finns i biblioteket” än faktisk produktförbättring.

Fix: byt follow-up-kedjans classifier till fallback-varianten, men behåll no-key-paritet.

2. LLM-fallback kan semantiskt slå på preview för fel messageKind

Koden omräknar shouldStartPreview enbart från buildRequirement.

Men RouterDecision har ingen cross-field-validering som säger “answer_only får inte ha targeted_rebuild”. Den definierar fälten, men saknar sådan semantisk validator.

Påverkan: om modellen returnerar schema-giltigt men semantiskt inkonsekvent beslut, till exempel messageKind="answer_only" + buildRequirement="targeted_rebuild", kan preview flaggas trots att det är en fråga.

Fix: normalisera/avvisa inkonsekventa kombinationer efter LLM-svar:

messageKind	Tillåtet buildRequirement
answer_only	none
component_discovery	none
site_review	none eller plan_only
reference_analysis	plan_only
bug_report	plan_only
unclear	none
edit_instruction	artifact_patch_only / targeted_rebuild / full_rebuild
multi_intent	plan_only / targeted_rebuild / full_rebuild
3. OpenClaw orchestrate() använder inte LLM-fallback

OpenClaw Core importerar classify_message och använder den direkt.

Påverkan: den nya fina OpenClaw-ingången går runt KÖR-6b. Det är inte katastrof eftersom decide(router, context) kan få en färdig router-decision från annan caller, men raw-message-vägen blir inte “senaste router”.

Fix: orchestrate() bör antingen:

använda classify_message_with_llm_fallback, eller
acceptera en classifier= dependency injection (funktion som kan bytas i test/CLI/UI).
4. OpenClaw tappar externa referens-URL:er i context assembly

orchestrate() gör:

router = classify_message(...)
context = assemble_context(router.contextLevel, **context_kwargs)

men skickar inte router.reference.url som url.

assemble_context("external_reference") kräver url, annars returneras missing-context.

Påverkan: “gör något som aftonbladet.se” kan klassas rätt av routern men OpenClaw får inte rätt referenskontext. Då blir OpenClaw-planen ytlig eller felaktigt gated.

Fix: i orchestrate():

if router.reference:
    context_kwargs["url"] = router.reference.url
5. OpenClaw multi-intent kan bryta referens-säkerheten

Routern har särskild logik: multi-intent + extern referens ska bli plan_only, med reference och risk="do_not_copy_exact".

Men OpenClaw _multi_intent() gör patch_plan_request om någon subtask är edit, oavsett om routerbeslutet var reference-gated.

Påverkan: ett “lägg klocka som på X och ändra rubriken”-case kan gå mot patch istället för referensanalys först.

Fix: i _multi_intent():

if router.reference or router.buildRequirement == "plan_only" or router.risk == "do_not_copy_exact":
    return plan_only
6. OpenClaw säger fortfarande blockedBy="kor-7c" trots att kor-7c finns

Påverkan: OpenClaw-statusen är nu arkitektoniskt stale. Den borde inte säga att kor-7c saknas; den borde säga att OpenClaw action bridge saknas.

Fix: byt till något i stil med:

status: "action_bridge_missing"
blockedBy: "openclaw-action-bridge"
